API Gateway

API Gateway key points:

  • Create APIs as a front end access point for backend data and functionality.

  • Define traditional API methods such as: POST, GET, PUT/PATCH, DELETE (CRUD).

  • Stages allow for different published versions of an API to exist (i.e. - prod, dev, QA).

  • Every stage has a unique invoke url for the user to interact with the API.

  • API Gateway supports custom domains for API endpoints.

  • API needs to be published via 'Deploy API' in order for it to be accessed.

  • Resources are the URLs for an API (i.e. - /groups)

  • Resources can have child resources (i.e. - /groups/id/edit)

  • CORS (Cross Origin Resource Sharing) can be enabled on endpoints. CORS allows domains to utilize resources from exterior domains.

  • API caching improves latency and reduces amount of calls made to the API.

  • Same origin policies protect against XSS (Cross Site Scripting) attacks and ignore Postman/Curl requests. Same origin policies are the opposite of CORS.

  • Authorization to access API can be implemented with Cognito or Lambda.
